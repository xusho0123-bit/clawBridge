三步驟 API 流程3-Step API Flow

核心對話迴圈：

The core conversation loop:

1. StartCascade → 2. SendUserCascadeMessage → 3. GetCascadeTrajectory (poll)

1. StartCascade

建立新的對話 session，使用 UUID。

Creates a new conversation session with a UUID.

{
  "cascadeId": "uuid-here",
  "metadata": { "ideName": "antigravity" }
}

2. SendUserCascadeMessage

傳送使用者訊息。必須包含 cascadeConfig.plannerConfig.requestedModel。

Sends the user's message. Must include cascadeConfig.plannerConfig.requestedModel.

{
  "cascadeId": "uuid-here",
  "items": [{ "text": "Hello" }],
  "metadata": { "ideName": "antigravity" },
  "cascadeConfig": {
    "plannerConfig": {
      "conversational": {},
      "requestedModel": { "model": "MODEL_PLACEHOLDER_M18" }
    }
  }
}

3. GetCascadeTrajectory

輪詢 AI 回應。回傳 trajectory.steps[]，包含類型和狀態。

Polls for the AI response. Returns trajectory.steps[] with type and status.

{
  "cascadeId": "uuid-here",

}

透過分析 Language Server binary 的 protobuf 定義發現。SendUserCascadeMessageRequest 有一個 media 欄位用於 inline binary 資料。

Discovered by analyzing Language Server binary protobuf definitions via strings. The SendUserCascadeMessageRequest has a media field for inline binary data.

透過 media 欄位傳送圖片 / PDFSending Images / PDF via media field

{
  "cascadeId": "uuid-here",
  "items": [{ "text": "Describe this image" }],
  "media": [{
    "mimeType": "image/jpeg",
    "inlineData": "<base64-encoded-bytes>"
  }],
  "metadata": { "ideName": "antigravity" },
  "cascadeConfig": { ... }
}

軌跡結構Trajectory Structure VERIFIED

完整的 AI 回應軌跡依照以下步驟序列：

A complete AI response trajectory follows this step sequence:

USER_INPUT → CONVERSATION_HISTORY → EPHEMERAL_MESSAGE → PLANNER_RESPONSE → (tool calls) → PLANNER_RESPONSE → CHECKPOINT

已解決的問題Solved Problems

SOLVED

1. 重複回應Duplicate Responses

問題：Problem: 同一個 cascade 中多條訊息導致 pollResponse 回傳舊訊息的軌跡資料。Multiple messages in the same cascade caused pollResponse to return trajectory data from old messages.

解決：Solution: 快照式方法 — 發送前記錄 step count 和 executor count，只看快照之後的新步驟。Snapshot-based approach — record step count and executor count before sending, only look at new steps after the snapshot.

SOLVED

2. 回應過早截斷Premature Response Truncation

問題：Problem: 多步驟 AI 回應中，較早的 PLANNER_RESPONSE 步驟已標記 DONE，導致提前回傳。Multi-step AI responses had earlier PLANNER_RESPONSE steps marked DONE, causing early return.

解決：Solution: 只追蹤最後一個 PLANNER_RESPONSE 步驟的狀態。Track the status of the last PLANNER_RESPONSE step only.

SOLVED

3. AI 回應超時（Tool Calls）AI Response Timeout (Tool Calls)

問題：Problem: 複雜訊息觸發 tool calls，executorMetadatas 很慢才出現，導致 120 秒超時。Complex messages trigger tool calls; executorMetadatas is slow to appear, causing 120s timeout.

解決：Solution: 穩定性備援 — 如果最後的 PLANNER_RESPONSE 已 DONE 且文字穩定 5 秒，立即回傳。Stability fallback — if last PLANNER_RESPONSE is DONE and text stable for 5 seconds, return immediately.

SOLVED

4. AI 自動分析工作區AI Responds About Workspace

問題：Problem: AI 預設會分析工作區檔案，每次新對話都在總結專案進度。AI defaults to analyzing workspace files, summarizing project progress on every new conversation.

解決：Solution: 系統提示詞注入 — 在每個 cascade 的第一則訊息前加上 [System Instructions]。可透過 .env 的 SYSTEM_PROMPT 設定。System Prompt injection — prepend [System Instructions] to the first message of each cascade. Configurable via SYSTEM_PROMPT in .env.

SOLVED

5. Telegram 409 Conflict

問題：Problem: 多個 Bot 實例使用相同 token 造成 409 衝突。Multiple bot instances using the same token cause 409 Conflict errors.

解決：Solution: start.command 啟動前先殺掉舊的 bridge 程序。start.command kills old bridge processes before starting.

SOLVED

6. IDE 重啟自動重連Auto-Reconnect on IDE Restart

問題：Problem: IDE 重啟或切換專案會改變 LS port/CSRF。IDE restart or project switch changes LS port/CSRF.

解決：Solution: 多層恢復機制：Multi-layer recovery:

  • telegram.js: ECONNREFUSED → 自動呼叫auto-call reconnectFn()
  • rpc.js: cascade 錯誤 → 自動重設 + 重試cascade error → auto reset + retry
  • 佇列卡住偵測：超時 + 60s → 強制重設Queue stuck detection: timeout + 60s → force reset
SOLVED

7. 串流更新Streaming Updates

問題：Problem: 長回應讓使用者等待時沒有回饋。Long AI responses leave user waiting with no feedback.

解決：Solution: 在 pollResponse 期間用 bot.editMessageText() 即時更新 TG 訊息。800ms 輪詢間隔，20 字最小差異閾值。During pollResponse, use bot.editMessageText() to update TG message in real-time. 800ms poll interval, 20-char minimum difference threshold.

SOLVED

8. 透過 Bridge 傳圖片辨識Image Recognition via Bridge

問題：Problem: TG 圖片下載到本地，文字路徑傳給 AI → AI 觸發 VIEW_FILE → 永遠卡住。IDE 內部貼圖正常。TG images downloaded locally, text path sent to AI → AI triggers VIEW_FILE → hangs forever. IDE internal image paste works fine.

解決：Solution: 透過 strings 分析 LS binary 發現 media 欄位。用 media: [{ mimeType, inlineData }] 直接傳 base64 圖片。AI 約 5 秒辨識，不需 tool call。Discovered media field in protobuf via strings analysis of LS binary. Using media: [{ mimeType, inlineData }] sends images as base64. AI recognizes in ~5s, no tool calls needed.

SOLVED

9. 語音輸入（STT）Voice Input (Speech-to-Text)

問題：Problem: 音訊透過 media 欄位傳送（API 回傳 OK），但 Gemini 模型透過 LS 無法處理 inline 語音內容。Audio data accepted via media field (API returns OK), but Gemini model through LS cannot process speech content inline.

解決：Solution: 使用 OpenAI Whisper 的本地 STT 管道。語音訊息：下載 OGG → ffmpeg 轉 WAV → Whisper 辨識 → 文字傳給 AI。Local STT pipeline using OpenAI Whisper. Voice messages: download OGG → ffmpeg convert to WAV → Whisper transcribe → text sent to AI.
